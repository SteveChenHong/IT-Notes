* 各自設計目的
* 行為差異（是否補跑、是否重疊）
* 正確使用情境
* 實務建議（工程師角度）
* 可直接複製使用的範例程式碼

## 一張總覽表（先看這張就好）

| 類型                       | 推薦程度      | 是否為新設計 | 是否 async 友善 | 是否可能重疊執行 | 是否補跑 Tick | 適合用途          |
| ------------------------ | --------- | ------ | ----------- | -------- | --------- | ------------- |
| `System.Timers.Timer`    | ❌ 不推薦新專案  | 舊      | ❌           | ✅ 可能     | ❌         | WinForm / 舊程式 |

> **99% 的後端 / Console / Background Service → 用 `PeriodicTimer`**

## 1️⃣ System.Timers.Timer

### 設計背景

* 為 **WinForm / WPF UI** 設計
* 事件驅動（`Elapsed` event）
* 實際上包裝了 `Threading.Timer`

### 核心特性

* 使用事件回呼（非 await）
* Tick 可能「重疊執行」
* 對 async/await 非常不友善

### 行為重點

```text
Tick 時間到 → 直接觸發 Elapsed
如果上一次還沒跑完 → 下一次照樣進來
```

### 範例（⚠️ 示意用，不推薦）

```csharp
var timer = new System.Timers.Timer(1000);

timer.Elapsed += (s, e) =>
{
    Console.WriteLine($"Tick {DateTime.Now}");
    Thread.Sleep(2000); // 模擬慢處理
};

timer.Start();
```

### 致命問題

* ❌ 容易產生重疊執行
* ❌ async void 地獄
* ❌ 難以控制生命週期

### 結論

> **新專案請完全避免使用**

---

## 2️⃣ System.Threading.Timer

### 設計背景

* **最底層 Timer**
* 給 framework / library / infra 用
* 無事件、無 async

### 核心特性

* 使用 callback
* 執行於 ThreadPool
* 不關心你有沒有跑完

### 行為模型

```text
時間到 → 丟進 ThreadPool
不等前一次 → 可能同時多個 callback
```

### 範例（低階控制）

```csharp
var timer = new System.Threading.Timer(_ =>
{
    Console.WriteLine($"Tick {DateTime.Now}");
    Thread.Sleep(2000);
}, null, 0, 1000);
```

### 優點

* 非常輕量
* 可完全控制 dueTime / period

### 缺點

* ❌ 極容易重疊
* ❌ 無 await
* ❌ 需自行處理 race condition

### 適合情境

* framework 開發
* 高頻、極低延遲系統
* 你 **100% 知道你在幹嘛**

---

## 3️⃣ ⭐ PeriodicTimer（強烈推薦）

### 設計背景

* .NET 6+ **現代 Timer 正解**
* 為 async/await 設計
* 解決所有舊 Timer 問題

### 核心特性

* `await timer.WaitForNextTickAsync()`
* **永遠不會重疊執行**
* 不補跑、不追趕時間

### 行為模型（超重要）

```text
等 1 秒 → 開始處理
處理 3 秒 → 再等 1 秒 → 下一次
（完全不追趕）
```

### 標準範例（完美 Timer）

```csharp
using var timer = new PeriodicTimer(TimeSpan.FromSeconds(1));

while (await timer.WaitForNextTickAsync())
{
    Console.WriteLine($"Tick {DateTime.Now}");
    await Task.Delay(2000); // 模擬慢處理
}
```

### 為什麼它是「完美 Timer」

* ✅ async 原生支援
* ✅ 不重疊
* ✅ 容易取消（CancellationToken）
* ✅ 行為可預期

## 關鍵差異：會不會「補跑」？

| Timer           | 若處理時間 > 間隔 | 行為       |
| --------------- | ---------- | -------- |
| Timers.Timer    | ❌ 不補       | 但會重疊     |
| Threading.Timer | ❌ 不補       | 但會重疊     |
| PeriodicTimer   | ❌ 不補       | **且不重疊** |

👉 **PeriodicTimer 是唯一「安全」的**

| Console App 計時            | `PeriodicTimer` |

## 最終結論（請記這段）

> **除非你在寫 framework 或 UI 老程式**
>
> **所有新專案，直接用 `PeriodicTimer`**

它是：

* 最安全
* 最好維護
* 最符合現代 .NET async 設計

## 延伸（你之前問過的問題對應）

* 為什麼 `PeriodicTimer` 不會「追回時間」？
  → 因為它是 **await 驅動，而不是事件排程**

* 為什麼 `System.Timers.Timer` 會出事？
  → 因為它是假定你處理很快（但現實不是）
